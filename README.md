# Simple C++ — Overview

## 1. Overview

Simple C++ S2S is a **source-to-source (S2S) multi-language compiler**.

It consists of:
- A **runtime library (C++)** responsible for:
  - type management
  - operator handling
  - casting
  - memory abstraction
- A **transcoder** that converts source languages into C++:
  - PHP (**in progress**)
  - Simple C++ (**later**)
  - JavaScript (**later**)
  - Python (**later**)

The transcoder converts each source language into C++ using a language-specific implementation.

### Compilation Pipeline

Source Language (SL)
→ SL Parser
→ SL AST
→ Transcoder (SL-specific implementation)
→ C++
→ Compiler
→ Binary

Each source language currently defines its own AST and transcoder.

A unified intermediate representation was evaluated but is not currently planned.
The majority of transformation logic is language-specific, making a shared IR
provide limited benefit relative to its complexity.

Each transcoder is responsible for producing valid C++ code that adheres to the runtime model and language constraints.

The generated C++ code is then compiled using a standard C++ toolchain.

---

## 2. Language Definition

Simple C++ is a **C++-inspired intermediate programming language** defined as:
- a **restricted and controlled subset of C++-like syntax**
- with **additional safety and usability constraints**

It is intended as a **controlled intermediate representation**, not a full C++ replacement.

Simple C++ can also be used directly as a source language in later stages.

---

## 3. Design Goals

### Core Principles
- Simplicity over completeness
- Readability over expressiveness
- Predictability over flexibility

### Objectives
- Maintain **low cognitive overhead**
- Be accessible to **junior developers**

### Safety
- No user-visible pointer semantics (only managed abstractions)
- Memory is **automatically managed**
- Out-of-bounds access is **checked**
- **Partial memory safety**, aiming to improve over time

### Interoperability
- Designed to interoperate with **C and C++**
- Leverages existing **C/C++ ecosystem**
- Native interop belongs to explicit C++ bridge/integration code, not to generated Simple C++ code

### Language Constraints
- No user-defined templates; limited built-in generic constructs are supported
- Limited and well-defined type system

---

## 4. Rationale

Simple C++ is designed to:
- remain easy to read and understand
- avoid complex ownership or lifetime systems
- avoid garbage collection
- leverage the existing C/C++ ecosystem

## 5. Implementation Strategy

### 5.1 Runtime Library

The Simple C++ runtime uses modern C++ features to enforce constraints.

The runtime library defines the core types and semantics used by generated code.

Runtime types in the `scpp` namespace are not user-facing and are generated by the transpiler.

Generated output must stay inside the Simple C++ semantic surface: no native primitive types, no direct `std::*`, and no direct use of native functions/classes as generated-language-visible constructs.

### 5.2 Memory Model

- No use of raw pointers
- Source-level `new` is translated into managed allocation and does not expose raw C++ allocation semantics
- All heap allocations are wrapped in managed constructs (e.g. shared ownership or unique ownership helpers)
- No user-visible pointer control

The runtime exposes:
- `create<T>()` → default ownership helper
- `shared<T>()` → explicit shared ownership helper
- `unique<T>()` → explicit unique ownership helper
- `weak(x)` → derive a non-owning reference from an existing owning value

The current default object-creation policy used by `create<T>()` is shared ownership.

### 5.3 Data Passing Rules

- Primitive types:
  - passed by value
  - returned by value

- `string`, `vector<T>`:
  - passed by `const &` (to avoid copies)

- Other types:
  - passed via managed pointer wrappers

### 5.4 Safety Guarantees

- Array indexing is **bounds-checked**
- Memory management currently relies on managed ownership, with shared ownership as the default object model

#### Limitations (Explicit)
- Does **not prevent reference cycles**; `weak_p` must be used correctly where needed
- Full memory safety is **not formally proven** at this stage
- Aliasing is currently unrestricted and relies on the managed ownership model

### 5.5 Code Generation Constraints

- Generated code is emitted inside `namespace scpp`
- Generated code must not access the C++ root namespace directly
- Generated values enter the runtime model explicitly

Examples:

    auto x = (int_t)12;
    auto s = (string_t)"text";
    auto a = create<MyClass>();
    auto b = shared<MyClass>();
    auto c = unique<MyClass>();
    auto w = weak(a);

## 6. Status Notes

- Ownership model: **work in progress**
- `create<T>()` currently defaults to shared ownership
- C/C++ interoperability details: **work in progress**

---

## 7. Non-Goals

- Full C++ compatibility
- Zero-cost abstractions at all costs
- Complete memory safety guarantees at this stage
- Copy-paste compatibility with source languages
